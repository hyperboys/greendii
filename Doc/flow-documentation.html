3. สถานะเอกสาร (DocStatus)

⬜ draft — แบบร่าง / ยังไม่ส่ง
🟡 pending — รออนุมัติ (อยู่ในสายอนุมัติ)
🟢 approved — อนุมัติแล้ว
🔴 rejected — ถูกปฏิเสธ (แก้ไขแล้วส่งใหม่ได้)
⚫ cancelled — ยกเลิก (ลบแบบ soft-delete)
⚠️
ลบ/ยกเลิกได้เฉพาะเอกสารที่อยู่ในสถานะ draft หรือ rejected เท่านั้น
WorkOrder ที่ถูก approved ครั้งสุดท้ายจะถูก set isClosed = true และมี closedAt

4. Flow — ใบเสนอราคา (Quotation)

1
👤 เซลล์ (Sales)

สร้างใบเสนอราคา

  • กรอกชื่อลูกค้า, โครงการ, รายการสินค้า/บริการ (ราคาวัสดุ + ค่าแรง)
  • ระบุส่วนลดพิเศษ, VAT, Grand Total
  • รหัสเอกสารออกอัตโนมัติ: QGD-MMYY-INITIALS+SEQ (เช่น QGD-0526-KC001)
  • สถานะเริ่มต้น: draft
2
👤 เซลล์ (Sales)

ส่งเข้าอนุมัติ (Submit)

  • กดปุ่ม "ส่งอนุมัติ" → status เปลี่ยนเป็น pending
  • ระบบดึง firstStep จาก approvalFlowConfig (default: Step 1 = เซลล์คนอื่น)
  • ส่ง LINE Notification แจ้ง approver ขั้นแรก
  • บันทึก ApprovalLog action = submit
3
🔄 Approver ตามสายอนุมัติ

สายอนุมัติ (Multi-step Approval)

  • Default flow: Step 1 เซลล์อื่น Step 2 ผจก.ขาย Step 3 ผจก.บริหาร Step 4 ผจก.โครงการ Step 5 กรรมการ
  • แต่ละขั้น: approver กด อนุมัติ → ระบบหา nextStep → notify ขั้นถัดไป
  • ถ้า reject → status = rejected, แจ้ง Sales ให้แก้ไข
  • ถ้าเป็น Step สุดท้าย → status = approved, แจ้ง Sales
4
✅ ระบบ

อนุมัติสมบูรณ์ / ปฏิเสธ

  • Approved: เอกสารพร้อมนำไปสร้าง Work Order ต่อได้
  • Rejected: Sales แก้ไขและส่งใหม่ได้ (status กลับเป็น draft หลังแก้ไข)

5. Flow — ใบสั่งงาน (Work Order)

1
👤 เซลล์ (Sales)

สร้างใบสั่งงาน

  • เลือก Quotation อ้างอิง (optional) + กรอกโครงการ, ที่ตั้ง, สินค้า
  • กำหนดทีมงาน, วัน QC, วันติดตั้ง, รายการ Checklist เอกสาร
  • รหัส: WOYYNNN เช่น WO26001
  • status เริ่มต้น: draft
2
👤 เซลล์ (Sales)

ส่งเข้าอนุมัติ

  • Default flow: Step 3 ผจก.บริหาร Step 4 ผจก.โครงการ Step 5 กรรมการ
  • ส่ง LINE Notification แจ้งผู้อนุมัติขั้นแรก
3
🔄 Approver ตามสายอนุมัติ

อนุมัติแบบ Multi-step

  • เหมือน Quotation — เดินสายจนครบทุก Step
  • Step สุดท้าย approve → status = approved + isClosed = true + closedAt = now()
  • WO ที่ closed แล้ว ไม่สามารถแก้ไขได้อีก
4
👤 เซลล์ / ทีมงาน

ดำเนินงานและสร้างเอกสารต่อเนื่อง

  • สร้าง ใบขอซื้อ (PR) เพื่อจัดหาวัสดุ
  • สร้าง ใบส่งมอบงาน (Handover) หลังงานเสร็จ

6. Flow — ใบขอซื้อ (Purchase Request)

1
👤 เซลล์ (Sales)

สร้างใบขอซื้อ

  • เลือก Work Order อ้างอิง (optional) + กรอกรายการสินค้าที่ต้องการจัดซื้อ
  • ระบุ Part No, รายละเอียด, จำนวน, ราคา, วันที่ต้องการ
  • รหัส: PRYYNNN เช่น PR26001
2
🔄 Approver ตามสายอนุมัติ

ส่งเข้าอนุมัติ + สายอนุมัติ

  • Default flow: Step 3 ผจก.บริหาร Step 4 ผจก.โครงการ Step 5 กรรมการ Step 6 จัดซื้อ
  • ขั้นตอน Procurement (Step 6) รับเรื่องและดำเนินการจัดซื้อ
3
✅ ฝ่ายจัดซื้อ (Procurement)

อนุมัติสมบูรณ์

  • ฝ่ายจัดซื้อเป็น Approver ขั้นสุดท้ายใน default flow
  • หลัง approved → ฝ่ายจัดซื้อดำเนินการสั่งซื้อต่อ (นอกระบบ)

7. Flow — ใบส่งมอบงาน (Hand Over Job)

1
👤 เซลล์ (Sales)

สร้างใบส่งมอบงาน

  • เลือก Work Order อ้างอิง + กรอกข้อมูลผู้รับมอบ, ผู้รับเหมา, สถานที่
  • ประเมินคะแนน: คุณภาพสินค้า / การขาย / การติดตั้ง (0–5)
  • บันทึก Comment จากลูกค้า
  • รหัส: HOYYNNN เช่น HO26001
2
🔄 Approver ตามสายอนุมัติ

ส่งเข้าอนุมัติ + สายอนุมัติ

  • Default flow: Step 3 ผจก.บริหาร Step 4 ผจก.โครงการ Step 5 กรรมการ
3
✅ ระบบ

อนุมัติสมบูรณ์ — งานปิดสมบูรณ์

  • งานโครงการนั้นถือว่าปิดสมบูรณ์แล้ว
  • ข้อมูลคะแนนความพึงพอใจถูกบันทึกเพื่อใช้ใน Reports

8. สายการอนุมัติ (Approval Flow)

ประเภทเอกสาร สายอนุมัติ Default หมายเหตุ
📋 ใบเสนอราคา 1 เซลล์อื่น 2 ผจก.ขาย 3 ผจก.บริหาร 4 ผจก.โครงการ 5 กรรมการ Step 1 ต้องเป็น Sales ที่ไม่ใช่ผู้สร้าง
🔧 ใบสั่งงาน 3 ผจก.บริหาร 4 ผจก.โครงการ 5 กรรมการ Step สุดท้าย → WO closed
🛒 ใบขอซื้อ 3 ผจก.บริหาร 4 ผจก.โครงการ 5 กรรมการ 6 จัดซื้อ ฝ่ายจัดซื้อเป็น Approver ขั้นสุดท้าย
✅ ใบส่งมอบ 3 ผจก.บริหาร 4 ผจก.โครงการ 5 กรรมการ ปรับได้จากหน้า Admin → สายอนุมัติ
⚙️
ปรับสายอนุมัติได้จาก: Admin → สายการอนุมัติ
ค่า config เก็บใน settings.approvalFlowConfig (JSON) ใน Database
ตัวอย่าง: { "quotation": [1,2,3,4,5], "workOrder": [3,4,5], "pr": [3,4,5,6], "handover": [3,4,5] }

10. บทบาทผู้ใช้ (User Roles)

Roleชื่อสิทธิ์หลัก
admin System Admin จัดการผู้ใช้, ตั้งค่าระบบ, ดูข้อมูลทุกอย่าง, ไม่อยู่ในสายอนุมัติ
sales เซลล์ สร้างและจัดการเอกสารของตัวเอง, อนุมัติขั้น Step 1 ของคนอื่น
sale_mgr ผู้จัดการฝ่ายขาย ดูเอกสารทีมขายทุกคน, อนุมัติ Step 2, เข้าถึง Reports
admin_mgr ผู้จัดการฝ่ายบริหาร อนุมัติ Step 3, ดูข้อมูลทุกเอกสาร, เข้าถึง Reports
project_mgr ผู้จัดการโครงการ อนุมัติ Step 4, ดูงานทุกโครงการ
director กรรมการผู้จัดการ อนุมัติ Step 5 (สูงสุด), ดูข้อมูลทุกอย่าง, Reports ทุกประเภท
procurement จัดซื้อ อนุมัติ Step 6, ดูใบขอซื้อทั้งหมด
factory โรงงาน/ผลิต ดูใบสั่งงานทั้งหมด, อนุมัติ Step 7 (ถ้ากำหนด)

11. การแจ้งเตือน (LINE Notification)

📤 Submit
→
แจ้ง Approver ขั้นแรก: "เอกสาร [DocNo] รอการอนุมัติจากคุณ"
✅ Approve (ไม่ใช่ขั้นสุดท้าย)
→
แจ้ง Approver ขั้นถัดไป: "เอกสาร [DocNo] รอการอนุมัติจากคุณ"
🎉 Approve (ขั้นสุดท้าย)
→
แจ้ง Sales เจ้าของเอกสาร: "เอกสาร [DocNo] ได้รับการอนุมัติแล้ว"
❌ Reject
→
แจ้ง Sales เจ้าของเอกสาร: "เอกสาร [DocNo] ถูกปฏิเสธ"
📱
การแจ้งเตือนใช้ LINE Messaging API — ผู้ใช้ต้องผูก lineUserId ในโปรไฟล์
ระบบ notify แบบ fire-and-forget (.catch(() => {})) — ถ้า LINE ล้มเหลวจะไม่กระทบการอนุมัติ

12. API Endpoints หลัก

📋 Quotations /api/quotations

  • GET /api/quotations รายการ (filter: status, salesId, q)
  • GET /api/quotations/:id รายละเอียดพร้อม items, approvalLogs, attachments
  • POST /api/quotations สร้างใหม่ (status: draft)
  • PUT /api/quotations/:id แก้ไข (เฉพาะ draft/rejected)
  • POST /api/quotations/:id/submit ส่งเข้าอนุมัติ → pending
  • POST /api/quotations/:id/approveอนุมัติ (ขยับ step หรือ approved)
  • POST /api/quotations/:id/reject ปฏิเสธ → rejected
  • DEL /api/quotations/:id ยกเลิก (เฉพาะ draft/rejected)

🔧 Work Orders /api/workorders

  • GET /api/workorders รายการ (filter: status, salesId, isClosed, q)
  • GET /api/workorders/:id รายละเอียด
  • POST /api/workorders สร้างใหม่
  • PUT /api/workorders/:id แก้ไข (ถ้า isClosed จะ error)
  • POST /api/workorders/:id/submit ส่งอนุมัติ
  • POST /api/workorders/:id/approve อนุมัติ (ขั้นสุดท้าย → isClosed=true)
  • POST /api/workorders/:id/reject ปฏิเสธ
  • DEL /api/workorders/:id ยกเลิก (เฉพาะ draft/rejected)

🛒 Purchase Requests /api/pr

  • GET /api/pr รายการ
  • GET /api/pr/:id รายละเอียด
  • POST /api/pr สร้างใหม่
  • PUT /api/pr/:id แก้ไข
  • POST /api/pr/:id/submit ส่งอนุมัติ
  • POST /api/pr/:id/approve อนุมัติ
  • POST /api/pr/:id/reject ปฏิเสธ
  • DEL /api/pr/:id ยกเลิก

✅ Handovers /api/handovers

  • GET /api/handovers รายการ
  • GET /api/handovers/:id รายละเอียด + คะแนนประเมิน
  • POST /api/handovers สร้างใหม่ + ประเมินคะแนน 3 ด้าน
  • PUT /api/handovers/:id แก้ไข (เฉพาะ draft/rejected)
  • POST /api/handovers/:id/submit ส่งอนุมัติ
  • POST /api/handovers/:id/approve อนุมัติ
  • POST /api/handovers/:id/reject ปฏิเสธ
  • DEL /api/handovers/:id ยกเลิก
